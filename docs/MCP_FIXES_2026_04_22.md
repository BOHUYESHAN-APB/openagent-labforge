# MCP 修复总结 - 2026-04-22

## 修复的问题

### 1. MCP 自动启用问题
**问题**：从 `.mcp.json` 加载的用户配置 MCP 被强制启用，忽略 `disabled` 字段

**根本原因**：`src/features/claude-code-mcp-loader/transformer.ts` 硬编码 `enabled: true`

**修复**：
```typescript
// 修复前
enabled: true  // 总是 true

// 修复后
const enabled = expanded.disabled !== true  // 尊重 disabled 字段
```

**影响**：只影响用户自己配置的 MCP，不影响插件内建的 4 个默认启用的 MCP

---

### 2. semantic_scholar_fastmcp 端口冲突问题

**问题**：
- `semantic_scholar_fastmcp` 启动时在 8000 端口开启 HTTP bridge
- 多个实例会导致端口冲突：`[Errno 10048] error while attempting to bind on address ('0.0.0.0', 8000)`
- 导致 MCP 启动失败：`Connection closed local mcp startup failed`

**根本原因**：
1. FastMCP 默认在 8000 端口启动 HTTP bridge（即使只需要 stdio）
2. 旧实例未正确关闭，占用端口
3. 8000 是常用端口，容易冲突

**修复**：
```typescript
export const semantic_scholar_fastmcp: LocalMcpConfig = {
  type: "local",
  command: ["uvx", "--native-tls", "--from", "semantic-scholar-fastmcp<3.0", "semantic-scholar-fastmcp"],
  enabled: false,
  environment: {
    FASTMCP_HTTP_PORT: "58473",  // 使用不常见端口避免冲突
  },
  timeout: LOCAL_MCP_STARTUP_TIMEOUT_MS,
}
```

**端口选择**：
- **58473** - 在动态/私有端口范围 (49152-65535)
- 不常用，不易冲突
- 如果需要禁用 HTTP bridge，可以设置 `FASTMCP_DISABLE_HTTP: "true"`

**版本固定**：
- 固定到 `semantic-scholar-fastmcp<3.0`
- FastMCP 3.0 有破坏性变更
- 日志显示：`✨ FastMCP 3.0 is coming! Pin fastmcp < 3 in production`

## MCP 传输协议说明

- 使用 HTTPS 连接到外部服务器

### 本地 MCP（stdio 传输）
- **paper_search_mcp**, **open_websearch_mcp**, **arxiv_mcp**
- 纯 stdio 传输，不开启 HTTP 服务
- 不占用端口
- 无端口冲突问题

### 本地 MCP（stdio + HTTP bridge）
- **semantic_scholar_fastmcp**
- 主要使用 stdio 传输（正常工作）
- 额外开启 HTTP bridge（可选，用于调试）
- ⚠️ 需要配置端口避免冲突

## 端口分配策略

### 原则
1. **优先使用纯 stdio** - 不需要 HTTP 端口
2. **避免常用端口** - 8000, 8080, 3000 等容易冲突
3. **使用动态端口范围** - 49152-65535
4. **每个 MCP 独立端口** - 避免相互冲突

### 当前分配
- **semantic_scholar_fastmcp**: 58473

### 未来扩展
如果有其他基于 FastMCP 的 MCP 需要 HTTP bridge：
- 58474, 58475, 58476...（递增分配）

## paper_search_mcp 网络访问问题

### 问题现象

paper_search_mcp 的三个工具可能遇到网络连接超时：

- `paper_search_mcp_search_pubmed` - 连接 pubmed.ncbi.nlm.nih.gov 超时
- `paper_search_mcp_search_biorxiv` - 连接 biorxiv.org 超时  
- `paper_search_mcp_search_google_scholar` - 连接 scholar.google.com 超时

### 错误日志示例

```
ERROR Search error: HTTPSConnectionPool(host='scholar.google.com', port=443): 
Max retries exceeded with url: /scholar?q=...
(Caused by ConnectTimeoutError(...'Connection to scholar.google.com timed out'))
```

### 根本原因

这**不是 MCP 配置问题**，而是**网络访问限制**：

1. Google Scholar 在某些地区可能被限制访问
2. 学术数据库可能需要特定网络环境
3. 防火墙或企业网络可能阻止直接访问

### 解决方案

如果你需要通过代理访问，在 `src/mcp/extended.ts` 中配置：

```typescript
export const paper_search_mcp: LocalMcpConfig = {
  type: "local",
  command: ["uvx", "--native-tls", "--from", "paper-search-mcp", "python", "-m", "paper_search_mcp.server"],
  enabled: true,
  timeout: LOCAL_MCP_STARTUP_TIMEOUT_MS,
  environment: {
    HTTP_PROXY: "http://your-proxy:port",
    HTTPS_PROXY: "http://your-proxy:port",
    // NO_PROXY: "localhost,127.0.0.1",  // 可选：排除本地地址
  },
}
```

### 测试网络连接

在配置代理前，可以先测试网络连接：

```bash
# 测试 Google Scholar
curl -I https://scholar.google.com

# 测试 PubMed
curl -I https://pubmed.ncbi.nlm.nih.gov

# 测试 BioRxiv
curl -I https://biorxiv.org
```

如果这些命令都超时，说明需要配置代理或使用 VPN。
